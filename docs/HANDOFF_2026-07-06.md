### Batch mới nhất — 0064 Secure push sender (tóm tắt)
Trigger `tg_push_on_notify()` giờ đọc secret `push_webhook_secret` từ **Vault** (fallback `current_setting('app.push_webhook_secret')`); nếu **không có secret → bỏ qua** gọi webhook (raise warning) thay vì gọi Edge Function không xác thực. Khi gọi thì gửi header `x-push-webhook-secret`. Edge Function `send-push` kiểm tra header này. → Cần đặt secret ở cả DB (Vault) và Edge Function để push hoạt động.

## 8. Kết luận review kiến trúc — 2026-07-07
Kết luận ngắn gọn: hướng chính của app vẫn đúng. Mô hình **RPC-first** với RLS deny-by-default là xương sống phù hợp cho phần mềm nội bộ này, nhất là vì business rules nằm gần dữ liệu và server vẫn là nơi chốt quyền.

Nhưng repo đã vượt qua giai đoạn "một file HTML lớn + nhiều migration override" một cách thoải mái. Trước khi redesign lớn, nên xử lý các điểm sau theo slice nhỏ:

1. **Siết RPC chi tiết công nợ trước.**
   - `rpc_get_debt_detail(p_ma_cn text)` hiện chỉ kiểm tra `auth.uid()` rồi trả nhiều dữ liệu nhạy cảm: số tài khoản NCC, MST, chứng từ, báo giá, số dư, lịch sử duyệt.
   - Cần thêm migration mới override RPC này để dùng `require_permission(...)` hoặc helper quyền riêng kiểu `can_view_debt_detail_(actor, debt)`.
   - Không sửa migration cũ.

2. **Gỡ hoặc vô hiệu hóa RPC chi tiền batch cũ.**
   - UI hiện dùng `rpc_cashier_pay_line(...)` cho Thủ quỹ chi từng dòng, có khóa dòng, kiểm tra số dư, ảnh chứng từ từng khoản, số tiền thực chi.
   - `rpc_execute_payment_request(text, jsonb)` vẫn còn callable từ migration cũ và có thể mutate payments/debts theo đường khác, không đi qua các invariant mới.
   - Nên thêm migration mới `drop function if exists rpc_execute_payment_request(text, jsonb);` / revoke rõ ràng, hoặc làm nó delegate an toàn qua module hiện hành.

3. **Frontend cần deepening dần, không rewrite ồ ạt.**
   - `public/index.html` đang trộn routing, permission UI, render HTML, parse form, validate, upload file, gọi RPC, và rule nghiệp vụ.
   - Nên tách theo module sâu: screen registry/navigation; proposal form payload; debt/payment detail; Excel/YCTT export.
   - Mục tiêu là tăng locality: sửa một workflow không phải đụng nhiều vùng rải rác trong file.

4. **Cần tài liệu "active RPC map".**
   - Vì quy ước đúng là không sửa migration cũ mà override bằng migration mới, lịch sử hiện có nhiều definition cho cùng RPC.
   - Nên tạo tool/tài liệu liệt kê "RPC đang active nằm ở migration nào" để review và merge an toàn hơn.

5. **Test surface nên đi qua interface thật.**
   - `supabase/tests/ap_flow_simulation.sql` là hướng tốt vì gọi RPC thật.
   - Nên bổ sung case cho: quyền xem `rpc_get_debt_detail`, stale cashier RPC, chi từng dòng chống race/vượt số dư, và dữ liệu chứng từ riêng tư.

Ưu tiên đề xuất: làm hai fix SQL đầu tiên trước khi redesign frontend. Đây là vùng rủi ro bảo mật/trạng thái cao nhất và phù hợp với cách làm slice-based của repo.
